# ADR-0008: Structured Logging Boundary Policy

## Status

Accepted — Phase 1

## Context

Logging is essential for observability and incident replay. However, traditional logging
patterns (`logger.info(...)` inside business logic) violate two Aegis core constraints:

1. **No side effects inside the core pipeline** (ADR-0006): a log flush is a side effect.
2. **Determinism**: log handler state (buffering, destination, level) is environmental
   state the core must not depend on.

The design question was: how does Aegis make logging useful without corrupting pipeline
purity?

## Decision

The Aegis logging model separates **log event construction** from **log event emission**.

**Inside the core pipeline:**
- Functions may construct `AegisLogEvent` value objects (frozen dataclasses).
- `AegisLogEvent` carries structured metadata: `event_type`, `layer`, `outcome`,
  optional `audit_id`, `plan_id`, `reason`, `metadata` mapping.
- `AegisLogEvent` does **not** carry a timestamp generated by the core. Timestamps are
  supplied by the outer adapter if needed.
- No log handler is called, configured, or imported inside pure pipeline functions.

**Outside the core (adapters, CLI, scenario runner):**
- Outer adapters receive `AegisLogEvent` objects and emit them to a sink.
- `structlog` may be used at the adapter boundary.
- Timestamps are added by the adapter at emission time.

**`src/aegis/aegis_logging.py`** provides:
- The `AegisLogEvent` frozen dataclass.
- `make_log_event(...)` — a pure constructor helper.
- `serialise_log_event(...)` — converts an event to a JSON-compatible dict.
- No global logger, no `logging.basicConfig()`, no handler setup.

## Consequences

**Positive:**
- Pipeline functions are still testable in complete isolation with no log-capture fixtures.
- Log events are value objects — they can be asserted in tests, diffed, and replayed.
- `structlog` or any other sink can be swapped at the adapter boundary without touching
  the core.
- Log event data is part of the audit trail: the structured fields mirror the audit
  contract fields, making log-based replay consistent with audit-based replay.

**Negative:**
- Developers must return or yield log events rather than calling `logger.info()` inline.
  This requires an adjustment for developers accustomed to traditional logging.
- Phase 1 pipelines do not yet thread log events out; that is a near-term adapter task.

## Alternatives Considered

**`structlog` configured globally at module import:** Rejected. Module-level logger
configuration is a side effect at import time. It also creates test isolation problems
when different tests want different log levels.

**Python stdlib `logging` module inside core functions:** Rejected. `logging.getLogger()`
is a global registry lookup — a hidden side effect. Log handler state is environmental.

**No logging at all in Phase 1:** Considered but rejected. The `aegis.aegis_logging` module
is referenced as canonical in documentation. Leaving it absent creates repo-contract
drift (the exact problem this ADR set is meant to resolve).
